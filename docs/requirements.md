# Claris 要件定義書

更新日: 2026-05-20  
対象: `Claris_app`

この文書は Claris の標準要件入口である。詳細な過去要件は `Claris_requirements_2026_05_18.md` を参照する。日付付き docs は詳細資料として残すが、今後の判断ではこの標準4ファイルを優先する。

## 1. 目的

Claris は、iPhone で日々のタスク、メモ、運営情報を素早く整理するための個人用 PWA とする。

データ同期 Phase の目的は、すぐに複数端末や共同編集を実装することではない。まず、ローカルデータ消失への備え、iPhone 内データのバックアップ、将来の複数端末対応、外部連携・AI 整理用 JSON 連携の土台を整える。

この Phase では既存 PWA 構成を維持し、まずローカルバックアップと同期用メタ情報をローカルデータへ追加する。サーバー同期 API や依存追加は、同期前に確定すべき要件と設計を docs に整理してから進める。

## 2. 現行構成

現行 Claris は次の構成で動作している。

- 静的 PWA
- HTML / CSS / JavaScript
- Service Worker
- IndexedDB
- JSON データ
- Node.js 標準 `http` サーバー
- iPhone での単体利用を主目的とするローカル保存アプリ

当面は次の方針を維持する。

- PWA として育てる。
- ローカル保存を主とする。
- オフラインでも使える状態を維持する。
- React / TypeScript / Vite へは現時点で移行しない。
- Express / SQLite / Drizzle ORM は、必要性が明確になるまで導入しない。
- Firebase / Supabase / PostgreSQL は使わない。

## 3. 現行機能要件

- 下部タブは「カレンダー」「今日」「一覧」の3タブ構成とする。
- アプリ起動後、初回だけカレンダータブを開いた時にローカル日付の今日を選択状態にする。ユーザーが別日を選択した後やタブ再訪時は、その選択状態を維持する。
- 開閉表示はアプリ全体で、表示する操作を `+`、非表示にする操作を `-` に統一する。
- タスク、メモ、運営情報を同じアプリ内で管理する。
- タスクは実施日、DL、優先度、任意の開始/終了時間、担当者、関連メモを持てる。
- タスク編集画面はタスク名の下に「日時」「資料」「その他」のタブを持ち、日時には実施日、DL、時間、優先度、繰り返し設定をまとめ、資料には関連メモと添付、その他には分類、担当者、補助項目を置く。タブ状態は一時的な UI 状態であり保存データに含めない。
- 優先タスクカードの色は、最優先を赤、2次優先を黄、3次優先を青とする。優先度の意味、順序、保存値は変更しない。
- 実施済みで DL 未到来のタスクカードは、右上表示を優先度ではなく「DL」とし、最優先表示と混同しない反転系デザインにする。
- 今日画面の優先タスクカードでも、サブタスクカードと同じ完了切替ボタンを表示し、完了/未完了の変更は通常のタスク更新と同じ `updatedAt`、`syncStatus`、`version` 更新経路を通す。
- タスクフォームの関連メモ選択は初期状態で閉じる。見出しは「関連メモ」とし、`+` で検索欄と一覧を表示、`-` で非表示にする。閉じた状態でも選択済み `task.memoIds` は保存時に失われない。
- メモは作成日と更新日を持ち、タイトル、本文、文字起こし、議題、方針、行動を扱う。
- メモ編集画面はタイトルの下に「本文」「録音」「内容整理」「その他」のタブを持つ。本文には手動入力欄、録音には録音操作と文字起こし欄、内容整理には議題、方針、行動、AI整理用JSON共有、AI整理結果取り込み、その他には添付資料、関連タスク、分類などを置く。タブ状態は保存データに含めない。
- メモ編集画面の文字起こし欄は、閉じている時に `+文字起こし`、開いている時に `-文字起こし` と表示する。
- メモ編集画面の関連タスク欄は一時的な UI 状態として開閉できる。閉じている時に `+関連タスク`、開いている時に `-関連タスク` と表示し、閉じた状態でも選択済み `memo.taskIds` とタスク側リンクを失わない。
- メモ編集画面の本文、議題、方針、行動の入力欄は右下の拡張ボタンで一時的に画面全体に近い高さまで大きく表示でき、再度押すと元に戻せる。拡張状態は保存データに含めない。
- タスク、メモ、運営情報には写真や資料を添付できる。初期実装では添付を IndexedDB のローカル state に保存し、`id`、`ownerType`、`ownerId`、`fileName`、`mimeType`、`size`、`createdAt`、`updatedAt`、`deletedAt`、`syncStatus`、`deviceId`、`version`、`dataUrl` または `blob` を保持する。
- 添付ファイルは大容量でアプリを壊さないよう初期上限を1ファイル8MBとし、削除は原則として `deletedAt` を使う論理削除にする。
- 日付、DL、保存、添付追加、添付削除、完了状態など、ユーザー操作で保存対象が変わる時は、操作内容が分かる短いトースト通知を表示する。
- 設定画面には外部 LLM 連携の入力欄を置かない。Claris 本体から外部 LLM へ直接 POST する設計は廃止する。
- メモ編集画面には「自動判定」ボタンを置かず、Claris 内では AI 処理やローカル自動分類を実行しない。
- 保存済みメモは本文と文字起こしを含む AI 整理用 `.json` ファイルを生成し、Web Share API 対応環境では iPhone の共有シートから外部 LLM アプリまたはファイル保存へ渡せる。未対応環境では `.json` ダウンロードへフォールバックする。
- AI 整理用 JSON には `clarisExportType`、`version`、`memoId`、`title`、`body`、`transcript`、`createdAt`、`updatedAt`、`instruction`、`expectedImportFormat` を含める。
- AI 整理結果は `.json` ファイル選択または JSON 文字列貼り付けで戻す「AI整理結果を取り込む」を持つ。
- AI整理結果の取り込みは `clarisImportType: "memo_ai_summary"`、`version: 1`、現在開いているメモと一致する `memoId`、`agendas` / `policies` / `actions` の文字列配列を検証し、不正な JSON、別メモ ID、配列でない値、文字列以外の配列要素では保存データを更新しない。
- AI整理結果の `title` は取り込み必須条件にしない。JSON の前後に説明文やコードブロック記号が混ざる場合は、安全に単一の JSON オブジェクトを抽出できる時だけ取り込む。前後の空白と BOM は除去してよいが、スマートクォートなど JSON として無効な文字は補正せずエラーにする。
- AI整理結果は `agendas` を内部キー `agenda`、`policies` を `decisions`、`actions` を `nextActions` へ反映する。表示ラベルは「議題」「方針」「行動」に統一し、内部キーは互換性維持のため変更しない。
- AI整理結果を反映する時は、議題の各行に `■`、方針の各行に `●`、行動の各行に `・` を付ける。文章は文末に `。` がなければ補い、単語だけの項目には無理に句点を付けない。
- 既存の「議題」「方針」「行動」がある場合は、AI整理結果の取り込み前に上書き確認を出す。取り込み成功後は通常のメモ更新と同じ経路で `updatedAt` と同期メタ情報を更新する。
- タスク、メモ、運営情報のカードを開いた状態でスクリーンショット、画面フォーカス変化、ビューポート変化が起きても、不要な再描画やカード内スクロール位置のリセットを起こさない。
- 運営情報はタイトル、種別、期間、分類、内容を持つ。
- 運営情報編集画面は「期間」「内容」「分類」「資料」のタブを持ち、期間には開始日と終了日、内容にはタイトルと本文、分類には種別と分類、資料には添付や関連資料を置く。タブ状態は保存データに含めない。
- 今日画面上部には「優先」「サブタスク」「メモ」「運営」のタブを置き、優先には優先タスクとDL超過タスク、サブタスクにはサブタスクカード、メモには今日に関係するメモ、運営には今日に関係する運営情報だけを表示する。今日画面のタブ状態は一時的な UI 状態とする。
- 削除済みタスク、メモ、運営情報、分類は削除済み一覧から復元できる。
- JSON / CSV / テキスト貼り付けインポートと JSON エクスポートを維持する。
- IndexedDB によるローカル保存を維持する。

## 5. 同期前バックアップ要件

同期処理を行う前に、必ずローカルデータのバックアップを作成する。

理由:

- 同期失敗によるデータ消失を防ぐため。
- サーバーデータによる誤上書きを防ぐため。
- 実装初期段階の不具合に備えるため。

バックアップには次を含める。

- バックアップ作成日時
- 対象データ全体
- データ件数
- アプリのデータ形式バージョン
- バックアップ作成理由: `manual` / `before-sync` / `before-restore` / `before-metadata-migration` / `scheduled`

保持方針:

- 最新1件だけでなく複数世代を保持する。
- 初期案として最大10件程度を保持する。
- 古いバックアップは自動削除してよい。
- 手動バックアップは自動削除対象から除外できる余地を残す。

復元要件:

1. 現在データを復元前バックアップとして保存する。
2. 選択したバックアップを復元する。
3. 復元日時を記録する。
4. 次回同期時にサーバーへ反映できる状態にする。

## 6. 同期対象データ

Claris 内でユーザーが作成・編集・削除するデータを同期対象とする。

対象:

- タスク
- メモ
- 運営情報、方針、施策
- 添付ファイル
- カテゴリ、分類、種別
- 表示順
- 完了状態
- 実施日、DL、期間、開始/終了時間
- 関連メモ、関連タスク
- 添付ファイルと添付先
- 作成日時、更新日時
- 削除状態

初期段階の対象外:

- Service Worker のキャッシュ
- アプリ本体ファイル
- アイコン
- 一時的な UI 状態
- 表示中タブなどの瞬間的状態

## 7. 同期用メタ情報

同期対象データには、最低限次のメタ情報を持たせる設計にする。

- `id`
- `createdAt`
- `updatedAt`
- `deletedAt`
- `syncStatus`
- `deviceId`
- `version`

`deletedAt`:

- 削除は物理削除ではなく、原則として論理削除を検討する。
- 削除情報をサーバーへ伝え、他端末にも反映しやすくする。
- 誤削除時に復元できる余地を残す。

`syncStatus` の初期案:

- `local-only`
- `pending`
- `synced`
- `conflict`

既存データは、サーバー同期 API 実装前の履歴データとして `local-only` を補完する。同期 API が存在しない段階で全件を `pending` にすると、将来の初回同期時に「明示的な未送信変更」と「移行で補完された既存データ」の区別がつきにくくなるためである。

新規作成、編集、削除、復元で変更されたデータは `pending` にする。削除状態は `syncStatus` の別値ではなく、`deletedAt` があることと `syncStatus: "pending"` の組み合わせで表す。

`version`:

- 既存データにない場合は `1` を補完する。
- 新規作成時は `1` とする。
- 編集、削除、復元など対象データを変更した時に `+1` する。

`deviceId`:

- 初回起動時に UUID 相当の値を生成し、`settings.deviceId` に保存する。
- 各同期対象データには最後に変更した端末として同じ `deviceId` を保持する。

`deletedItems`:

- 現行の復元機能を維持するため、削除時の退避先として残す。
- 削除された元データのスナップショットにも `deletedAt` を付与し、将来の同期では削除トゥームストーンとして扱える形にする。

## 8. 同期タイミング

同期は次のタイミングを想定する。

- アプリ起動時
- アプリ内で保存した時
- 一定時間ごと
- オンライン復帰時

オフライン時はローカル保存を優先する。同期に失敗してもユーザーの編集内容を消してはいけない。

## 9. 認証要件

passkey 認証は初期同期設計では必須にしない。

理由:

- 現在は1台運用が主目的である。
- まず同期前バックアップと同期の安全性を優先する。
- 認証実装を先に入れると複雑化するため。

将来的には passkey、ローカル端末識別、簡易 PIN、セッション管理を検討する。ただし初期同期設計では認証方式を固定しない。
