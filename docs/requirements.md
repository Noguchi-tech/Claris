# Claris 要件定義書

更新日: 2026-05-19  
対象: `Claris_app`

この文書は Claris の標準要件入口である。詳細な過去要件は `Claris_requirements_2026_05_18.md` を参照する。日付付き docs は詳細資料として残すが、今後の判断ではこの標準4ファイルを優先する。

## 1. 目的

Claris は、iPhone で日々のタスク、メモ、運営情報を素早く整理するための個人用 PWA とする。

データ同期 Phase の目的は、すぐに複数端末や共同編集を実装することではない。まず、ローカルデータ消失への備え、iPhone 内データのバックアップ、将来の複数端末対応、外部連携・LLM 連携の土台を整える。

この Phase では既存 PWA 構成を維持し、まずローカルバックアップと同期用メタ情報をローカルデータへ追加する。サーバー同期 API や依存追加は、同期前に確定すべき要件と設計を docs に整理してから進める。

## 2. 現行構成

現行 Claris は次の構成で動作している。

- 静的 PWA
- HTML / CSS / JavaScript
- Service Worker
- IndexedDB
- JSON データ
- Node.js 標準 `http` サーバー
- iPhone での単体利用を主目的とするローカル保存アプリ

当面は次の方針を維持する。

- PWA として育てる。
- ローカル保存を主とする。
- オフラインでも使える状態を維持する。
- React / TypeScript / Vite へは現時点で移行しない。
- Express / SQLite / Drizzle ORM は、必要性が明確になるまで導入しない。
- Firebase / Supabase / PostgreSQL は使わない。

## 3. 現行機能要件

- 下部タブは「カレンダー」「今日」「一覧」の3タブ構成とする。
- タスク、メモ、運営情報を同じアプリ内で管理する。
- タスクは実施日、DL、優先度、任意の開始/終了時間、担当者、関連メモを持てる。
- メモは作成日と更新日を持ち、タイトル、本文、文字起こし、議題、方針、行動を扱う。
- 保存済みメモは本文と文字起こしを外部 LLM へ手動貼り付けするための「AI整理用にコピー」と、LLM の JSON 回答を戻す「AI整理結果を取り込む」を持つ。
- AI整理結果の取り込みは `clarisImportType: "memo_ai_summary"`、`version: 1`、対象 `memoId`、`agendas` / `policies` / `actions` 配列を検証し、不正な JSON や別メモの結果では保存データを更新しない。
- 運営情報はタイトル、種別、期間、分類、内容を持つ。
- 削除済みタスク、メモ、運営情報、分類は削除済み一覧から復元できる。
- JSON / CSV / テキスト貼り付けインポートと JSON エクスポートを維持する。
- IndexedDB によるローカル保存を維持する。

7. 必要になった段階でサーバー実装
8. 認証強化

## 5. 同期前バックアップ要件

同期処理を行う前に、必ずローカルデータのバックアップを作成する。

理由:

- 同期失敗によるデータ消失を防ぐため。
- サーバーデータによる誤上書きを防ぐため。
- 実装初期段階の不具合に備えるため。

バックアップには次を含める。

- バックアップ作成日時
- 対象データ全体
- データ件数
- アプリのデータ形式バージョン
- バックアップ作成理由: `manual` / `before-sync` / `before-restore` / `before-metadata-migration` / `scheduled`

保持方針:

- 最新1件だけでなく複数世代を保持する。
- 初期案として最大10件程度を保持する。
- 古いバックアップは自動削除してよい。
- 手動バックアップは自動削除対象から除外できる余地を残す。

復元要件:

1. 現在データを復元前バックアップとして保存する。
2. 選択したバックアップを復元する。
3. 復元日時を記録する。
4. 次回同期時にサーバーへ反映できる状態にする。

## 6. 同期対象データ

Claris 内でユーザーが作成・編集・削除するデータを同期対象とする。

対象:

- タスク
- メモ
- 運営情報、方針、施策
- カテゴリ、分類、種別
- 表示順
- 完了状態
- 実施日、DL、期間、開始/終了時間
- 関連メモ、関連タスク
- 作成日時、更新日時
- 削除状態

初期段階の対象外:

- Service Worker のキャッシュ
- アプリ本体ファイル
- アイコン
- 一時的な UI 状態
- 表示中タブなどの瞬間的状態

## 7. 同期用メタ情報

同期対象データには、最低限次のメタ情報を持たせる設計にする。

- `id`
- `createdAt`
- `updatedAt`
- `deletedAt`
- `syncStatus`
- `deviceId`
- `version`

`deletedAt`:

- 削除は物理削除ではなく、原則として論理削除を検討する。
- 削除情報をサーバーへ伝え、他端末にも反映しやすくする。
- 誤削除時に復元できる余地を残す。

`syncStatus` の初期案:

- `local-only`
- `pending`
- `synced`
- `conflict`

既存データは、サーバー同期 API 実装前の履歴データとして `local-only` を補完する。同期 API が存在しない段階で全件を `pending` にすると、将来の初回同期時に「明示的な未送信変更」と「移行で補完された既存データ」の区別がつきにくくなるためである。

新規作成、編集、削除、復元で変更されたデータは `pending` にする。削除状態は `syncStatus` の別値ではなく、`deletedAt` があることと `syncStatus: "pending"` の組み合わせで表す。

`version`:

- 既存データにない場合は `1` を補完する。
- 新規作成時は `1` とする。
- 編集、削除、復元など対象データを変更した時に `+1` する。

`deviceId`:

- 初回起動時に UUID 相当の値を生成し、`settings.deviceId` に保存する。
- 各同期対象データには最後に変更した端末として同じ `deviceId` を保持する。

`deletedItems`:

- 現行の復元機能を維持するため、削除時の退避先として残す。
- 削除された元データのスナップショットにも `deletedAt` を付与し、将来の同期では削除トゥームストーンとして扱える形にする。

## 8. 同期タイミング

同期は次のタイミングを想定する。

- アプリ起動時
- アプリ内で保存した時
- 一定時間ごと
- オンライン復帰時

オフライン時はローカル保存を優先する。同期に失敗してもユーザーの編集内容を消してはいけない。

## 9. 認証要件

passkey 認証は初期同期設計では必須にしない。

理由:

- 現在は1台運用が主目的である。
- まず同期前バックアップと同期の安全性を優先する。
- 認証実装を先に入れると複雑化するため。

将来的には passkey、ローカル端末識別、簡易 PIN、セッション管理を検討する。ただし初期同期設計では認証方式を固定しない。
